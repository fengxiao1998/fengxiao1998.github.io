## 进程与线程



**进程**描述了 CPU 在运行指令及加载和保存上下文所需的时间，在应用层面来说就是一个程序，**线程**是进程中的更小单位，描述了执行一段指令所需的时间。进程——资源分配的最小单位，线程——程序执行的最小单位。



##### 单线程与多线程



- 单线程：单线程在程序执行时，所走的程序路径按照连续顺序排下来，前面的必须处理好，后面的才会执行。

- 多线程：多线程是指程序中包含多个执行流，即在一个程序中可以同时运行多个不同的线程来执行不同的任务，也就是说允许单个程序创建多个并行执行的线程来完成各自的任务。



##### JS单线程（原因及特点）



在浏览器中打开一个页面时，相当于创建了一个进程。这个进程中可以有多个线程，比如渲染线程、JS 引擎线程、HTTP 请求线程等等。当发起一个请求时，其实就是创建了一个线程，当请求结束后，该线程可能就会被销毁。



而其中提到的**JS 引擎线程**和**渲染线程**，众所周知，在 JS 运行的时候可能会阻止 UI 渲染，这说明了两个线程是互斥的。原因在于JS可以修改DOM，如果在JS执行的时候渲染线程也在工作的话，可能会导致不安全的渲染UI。单线程相对于多线程有着占用的内存较少，也能节约时间，便于控制的优点。



在不同的操作系统中，对于创建多进程所需的时间不同，对于Window系统来说，开启多进程需要的时间开销较大。而在Linux系统中，创建进程的时间开销较少，而某些服务器依靠大量创建进程工作，甚至对每个用户的请求就创建一个进程，因此服务器使用Linux系统要更为普遍。



## 执行栈

##### js中的堆（heap）内存与栈内存（stack）



heap是没有结构的，数据可以任意存放。heap用于复杂数据类型（引用类型）分配空间，例如数组对象、object对象。



stack是有结构的，每个区块按照一定次序存放（后进先出），stack中主要存放一些基本类型的变量和对象的引用，存在栈中的数据大小与生存期必须是确定的。可以明确知道每个区块的大小，因此，stack的寻址速度要快于heap。



程序运行时，每个线程分配一个stack，每个进程分配一个heap，也就是说，stack是线程独占的，heap是线程共用的。由于stack在创建的时候大小确定，所以当数据大小超出了stack的范围时，就会出现爆栈的问题。



##### 执行栈

![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/d20f44b5-746a-43bf-bf86-901f5a6cbf94.gif)



由上图，我们可以把执行栈理解成是一个存储函数调用的栈结构，遵循先进后出的原则。



当开始执行 JS 代码时，首先会执行一个 main 函数（*JS没有main函数，只是模拟出的main函数，方便理解*），然后执行代码。根据先进后出的原则，后执行的函数会先弹出栈，在图中我们也可以发现，foo 函数后执行，当执行完毕后就从栈中弹出了。



在代码执行时，可以利用断点调试查看代码执行过程中，调用栈的变化（可以自行搜索相关文档，例如https://segmentfault.com/a/1190000012646261）。或者在报错的函数中看一下执行栈的顺序



```
function foo() {
  throw new Error('error')
}
function bar() {
  foo()
}
bar()

//报错
VM89:2 Uncaught Error: error
    at foo (<anonymous>:2:9) // 报错在foo函数
    at bar (<anonymous>:5:3) // foo函数在bar中执行
    at <anonymous>:7:1
```



## Event Loop



![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/7ea7390f-29b1-45bd-8963-fa1ef3cf15a2.svg)



上面已经说过的堆（heap）和栈（stack）,在Event Loop中还有一个队列（queue）的概念。在执行同步代码的时候，就是往执行栈中放入函数。而在遇到异步代码时，就会被挂起，在需要执行的时候加入到Task队列中。当执行栈为空时，Event Loop就会从 Task 队列中拿出需要执行的代码并放入执行栈中执行，所以本质上来说 JS 中的异步还是同步行为。只不过是把异步代码放在Task队列中去等待执行而已。



引用一张比较经典的图来解释：

![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/dee16ba1-8f0b-4385-bd4d-547c291ffb68.png)



由图还能看出，不同的任务源会被分配到不同的任务对列，任务源可以分为 微任务（microtask） 和 宏任务（macrotask）。



**Event Loop 执行顺序**：



- 首先执行同步代码，这属于宏任务

- 当执行完所有同步代码后，执行栈为空，查询是否有异步代码需要执行

- 执行所有微任务

- 当执行完所有微任务后，如有必要会渲染页面

- 然后开始下一轮 Event Loop。



见下图，宏任务与微任务的执行顺序：

​    ![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/855b4c8f-5e98-4e86-a86d-7d427b57e159.png)



```
console.log('script start')

async function async1() {
  await async2()
  console.log('async1 end')
}
async function async2() {
  console.log('async2 end')
}
async1()

setTimeout(function() {
  console.log('setTimeout')
}, 0)

new Promise(resolve => {
  console.log('Promise')
  resolve()
})
  .then(function() {
    console.log('promise1')
  })
  .then(function() {
    console.log('promise2')
  })

console.log('script end')
```



虽然 setTimeout 写在 Promise 之前，但是因为 Promise 属于微任务而 setTimeout 属于宏任务，所以setTimeout的打印结果会在最后。



然而以上代码的运行结果在不同的环境中执行结果是不一样的，在node V8和Node V10的环境中执行结果分别：



```
script start>async2 end>Promise>script end>async1 end>promise1>promise2>setTimeout
script start>async2 end>Promise>script end>promise1>promise2>async1 end>setTimeout
```



在浏览器中的第一种结果，也就是await后的函数执行要快于promise的链式操作，这种其实是一种错误的规范。因为如果 await 后面跟着 Promise 的话，会在Promise 的链式操作全部完成后才执行async1 end，所以浏览器利用的Node 8的BUG，也算是优化了性能，将await提前了，目前已被同意这种做法。详情见解释https://github.com/tc39/ecma262/pull/1250



但在使用时应该注意相关情形，尤其是async和Promise混用时，参见https://v8.dev/blog/fast-async#await-under-the-hood。例如以下代码。



```
const p = Promise.resolve();

(async () => {
  await p; console.log('after:await');
})();

p.then(() => console.log('tick:a'))
  .then(() => console.log('tick:b'));
 
如果原意想在p函数全部执行完毕之后再去console.log('after:await')，此时浏览器就会将console.log('after:await')提前到tick:a和tick:b之前，这明显是一种错误的规范，因为同属于微任务，应该按照任务队列执行，所以Node V10中将此规范更改了回来。
```



微任务包括 process.nextTick ，promise ，MutationObserver。



宏任务包括 script ， setTimeout ，setInterval ，setImmediate ，I/O ，UI rendering。



## Node 中的 Event Loop



Node 中的 Event Loop 和浏览器中的是完全不相同的东西。



Node 的 Event Loop 分为 6 个阶段，它们会按照顺序反复运行。每当进入某一个阶段的时候，都会从对应的回调队列中取出函数去执行。当队列为空或者执行的回调函数数量到达系统设定的阈值，就会进入下一阶段。



![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/b7bac0ed-3005-4a65-8007-2f87b96618e7.png)

```
1. timers(计时器) ------- 执行setTimeout以及setInterval的回调
2. I/O callbcacks ------ 执行几乎所有的回调，除了close回调，timer的回调，和setImmediate()的回调
3. idle,prepare -------- node内部使用
4. poll(轮询) ----------- 获取新的I/O事件；node会在适当条件下阻塞在这里。
5. check --------------- 处理 setImmediate 回调
6. close callbcaks ----- 处理关闭的回调  如socke.on('close')
```



较为经典的例子如下，如果有兴趣可以自己去翻看相关资料：

```
setTimeout(() => {
    console.log('setTimeout')
}, 0)
setImmediate(() => {
    console.log('setImmediate')
})
在上述代码中，setTimeout 可能执行在前，也可能执行在后
·首先 setTimeout(fn, 0) === setTimeout(fn, 1)，这是由源码决定的
·进入事件循环也是需要成本的，如果在准备时候花费了大于 1ms 的时间，那么在 timer 阶段就会直接执行 setTimeout 回调
·那么如果准备时间花费小于 1ms，那么就是 setImmediate 回调先执行了
```



```
const fs = require('fs')

fs.readFile(__filename, () => {
    setTimeout(() => {
        console.log('timeout');
    }, 0)
    setImmediate(() => {
        console.log('immediate')
    })
})
在上述代码中，setImmediate 永远先执行。因为两个代码写在 IO 回调中，IO 回调是在 poll 阶段执行，当回调执行完毕后队列为空，发现存在 setImmediate 回调，所以就直接跳转到 check 阶段去执行回调了。
```



上面介绍的都是 macrotask 的执行情况，对于 microtask 来说，它会在以上每个阶段完成前清空 microtask 队列，下图中的 Tick 就代表了 microtask。

![img](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/ya2QnVEQdo4Jl4XB/img/b9acf340-5ec4-49ee-96b2-ff50c43c7b17.png)

```
setTimeout(() => {
  console.log('timer')
}, 0)

Promise.resolve().then(function() {
  console.log('promise')
})
以上代码来说，和浏览器中的输出是一样的，microtask 永远执行在 macrotask 前面。
```



还有一个特别的函数，即Node 中的 process.nextTick，这个函数其实是独立于 Event Loop 之外的，它有一个自己的队列，当每个阶段完成后，如果存在 nextTick 队列，就会清空队列中的所有回调函数，并且优先于其他 microtask 执行。



```
setTimeout(() => {
 console.log('timer1')

 Promise.resolve().then(function() {
   console.log('promise1')
 })
}, 0)

process.nextTick(() => {
 console.log('nextTick1')
 process.nextTick(() => {
   console.log('nextTick2')
   process.nextTick(() => {
     console.log('nextTick3')
     process.nextTick(() => {
       console.log('nextTick4')
     })
   })
 })
})
可以发现 nextTick 永远都是优先全部打印出来
```
